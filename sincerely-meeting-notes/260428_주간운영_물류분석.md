## 회의 내용 요약

### 1) TMS TO 수량 불일치 — 승승전기 건

WMS 출고 수량과 TMS TO(Transport Order) 수량 합산 간의 불일치가 발생했다. 승승전기 품목에서 티바 3개 분량이 더 빠져야 하는 상황인데, TMS로 꽂히는 수량이 실제 출고 수량과 일치하지 않는 구조적 문제가 드러났다. 현재는 전체 출고 수량만 맞으면 된다는 임시 운영 원칙을 적용 중이나, TO별 수량 정합은 별도로 처리해야 한다. 해결 방향은 WMS 출고 수량을 TO 3개로 쪼개어 수량 합산이 맞도록 조정한 뒤, 각 TO를 올바른 배송 TO에 연결하는 방식으로 정리하기로 합의했다.

### 2) Claude Code / AI 세팅 공유 — 경선님 온보딩

지수님이 구축한 Claude Code 환경(CLAUDE.md 기반 에이전트 하네스)을 경선님 PC에서도 동일하게 사용할 수 있도록 온보딩을 진행하기로 했다. 이를 위해 경선님 PC에 Claude Code를 설치하고, CLAUDE.md 설정 파일을 공유한다. 현재 Airtable 재고 베이스(자체 테스트 베이스 포함)를 동기화해 두었으며, 경선님이 셀파(Airtable 연동 시스템)와 분리된 분석 전용 베이스에서 자유롭게 분석할 수 있도록 권한을 부여한다. 아이디 발급 및 프로젝트 단위 오더 구조 베이스 사용 방향도 논의되었다.

### 3) WMS GAP 분석 현황

WMS 착수를 위한 사전 GAP 분석 작업을 진행 중이다. A테이블 기준으로 ILBS 스탠다드 대비 회계 정합성 점검을 시범 실시한 결과, SAP 기준으로 필요한 추가 테이블·필드값 항목들이 도출되었다. 로케이션 정보, 공급사 SLA 리드타임, 이국력사 등급(A/B/C등) 기준, Wave-Picking Task 테이블·필드 정의 등이 논의되었다. 현재까지 테이블 구조만 정의된 상태이며, AutoResearch는 미실행 상태다.

### 4) 피킹리스트 / 출고확인서 개선 — 라벨 이동 처리

출고확인서와 피킹리스트의 라벨 이동 방식을 재변경했다. 기존 방식에서 품목·수량·특수 수량 레이아웃이 개선되어, 스크바이 기준 98 단위로 품목과 수량이 한 화면에 명확히 출력되는 구조로 개선되었다. 이 개선에 대해 현장에서 매우 긍정적인 반응이 있었다. 다만 TMS 또는 셀포드(외부 시스템)와 연동 시 게이트 처리가 많아 라벨 1개당 약 20초 대기 시간이 발생하는 성능 이슈가 남아 있다.

### 5) 인수인계 문서 에어테이블 UI 전환

기존 Google Slides·Google Spreadsheet 기반으로 운영하던 인수인계 문서를 에어테이블 인터페이스로 전환했다. 각 담당자가 에어테이블 인터페이스에 직접 입력하는 구조로 설계하여, 보다 정형화된 인수인계 양식을 구현했다.

### GS1 관점 — 데이터 표준

GS1 표준 대비 현재 이슈를 점검하면 아래와 같다.

라벨 처리 시 셀포드(외부 TMS 시스템) 게이트를 1건씩 통과하는 구조는 GS1 SSCC(Serial Shipping Container Code) 기반 배치 스캔 원칙에서 벗어난다. GS1 기준에서는 출고 시 SSCC 단위로 패키지를 묶어 ASN(Advance Shipping Notice)을 발송하고, 배치 처리하는 것이 표준이다. 현재 1건당 20초 딜레이는 단건 API 호출 구조에서 발생하는 것으로, 배치 처리 또는 비동기 큐 방식으로 전환 시 해소 가능하다.

WMS GAP 분석에서 논의된 공급사 이국력사 A/B/C 등급은 GS1 GLN(Global Location Number) 기반 공급사 식별 체계와 연계하여 표준화하는 것이 중장기적으로 필요하다.

피킹리스트·출고확인서에 품목별 수량·특수 수량이 명시되는 방향으로 개선된 점은 GS1 레이블링 원칙(GTIN + 수량 + LOT 표기)에 부합하는 긍정적 방향이다.

## Recommendation From AI (물류 전문가 관점)

**물류 운영 전문가 (CLTD)**

TMS TO 수량 불일치 문제는 단순한 수동 조정 이슈가 아니라, WMS↔TMS 간 수량 동기화 프로세스 자체가 부재한 데서 비롯된다. 현장에서 "전체 합산만 맞으면 된다"는 임시 운영 기준은 단기적으로 작동할 수 있지만, 배송 오류나 클레임 발생 시 TO 레벨 추적이 불가능해 근본 원인 분석이 어려워진다. 즉시 해결책으로는 에어테이블에서 TO 수량 분할 시 반드시 합산 검증 로직을 자동화하고, 수동 조정 이력을 타임스탬프와 함께 로그로 남기는 것을 권장한다. 라벨 1건당 20초 딜레이는 현장 피킹 흐름을 심각하게 저해하므로, 셀포드 API 호출을 배치 처리 또는 비동기 방식으로 전환하는 것이 단기 내 필수 조치다.

**재고 관리 전문가 (CPIM)**

WMS GAP 분석에서 도출된 로케이션 마스터, 공급사 SLA 등급, Wave-Picking Task 구조는 CPIM 기준 재고 가시성의 핵심 요소다. 특히 로케이션 없는 피킹은 FIFO 원칙을 시스템적으로 보장할 수 없어, LOT 만료나 구형 재고 우선 출고 실패로 이어질 수 있다. AutoResearch를 이번 주 내 실행하여 현재 GAP 목록을 정식 문서화하고, 우선순위가 높은 로케이션 마스터 테이블부터 에어테이블에 구축하는 것을 권장한다. 공급사 SLA A/B/C 등급은 납기 지연 이슈와 직결되므로, 기준 정의 후 즉시 Airtable 마스터에 반영해야 한다.

**공급망 전문가 (CSCP)**

이번 회의에서 논의된 Claude Code 환경 공유와 경선님 분석 권한 부여는 단순한 도구 공유를 넘어, 조직 내 데이터 기반 의사결정 문화 확산의 시작점이다. E2E 공급망 가시성 확보를 위해서는 현재 지수님 1인이 관리하는 분석 파이프라인을 팀 전체가 활용할 수 있는 구조로 민주화하는 것이 중요하다. 중기적으로는 재고 베이스 단일화(베이스 통합 논의 방향) 및 프로젝트 오더 단위 분석 체계가 E2E 수요-재고-출고-배송 흐름을 하나의 뷰로 볼 수 있게 해준다. 모바일 자동화 구상은 현장 담당자의 즉각적인 오류 대응 능력을 높이는 전략적 가치가 있으므로 중기 로드맵에 포함할 것을 권장한다.

**시스템 구현 관점 (SAP/NestJS)**

TMS TO 수량 동기화 문제는 NestJS 전환 시 반드시 TO 생성 → 수량 확정 → WMS 연계의 트랜잭션 일관성을 보장하는 서비스 레이어로 구현해야 한다. SAP TM의 이동유형 601 기준처럼, 출고지시 확정 시 TO 수량이 잠기는 불변 원장 원칙을 NestJS + Supabase 구조에서도 INSERT ONLY 패턴으로 구현해야 한다. 라벨 처리 20초 딜레이는 현재 동기 API 호출 방식을 BullMQ 또는 유사 비동기 큐로 전환하면 해결 가능하다. WMS GAP 분석에서 도출된 로케이션·Wave·Task 테이블은 Supabase `wms` 스키마에 SAP EWM Storage Type→Section→Bin 계층을 그대로 매핑하는 설계를 권장한다.

**에어테이블 즉시 적용 방안**
1. 승승전기 TO 3개 분할: WMS 출고 수량 레코드에 TO 연결 필드 추가, 3개 TO에 수량 배분 후 합산 자동 검증 수식 추가
2. TO 수량 조정 이력 로그 테이블: 조정 일시, 조정 전/후 수량, 담당자 필드로 구성된 별도 로그 테이블 생성
3. 로케이션 마스터 테이블 초안 완성: Storage Type / Section / Bin 3계층 구조로 구성, 현재 창고 레이아웃 매핑
4. 공급사 SLA 등급 필드 추가: 공급사 마스터 테이블에 등급(A/B/C), 표준 리드타임, 실적 리드타임 필드 추가
5. 경선님 분석 전용 뷰 생성: 재고 베이스에 경선님 전용 필터 뷰 추가, 편집 권한 최소화(읽기+댓글 수준)
6. 피킹리스트 출고확인서 현장 피드백 수집: 현재 개선된 레이아웃으로 1주일 사용 후 현장 피드백 취합, 추가 개선 사항 정리
